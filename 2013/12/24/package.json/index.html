package.json翻译

Dec 24 2013

描述

这个文档列出package.json中要求的所有东西，都是需要你去了解的，它一定是一个真实的json, 而不只是一个js对象字面量

这个文档中描述的大部分行为都被npm-config(7)描述的配置设置所影响

默认值

npm会有一些基于包内容的默认值

  • "scripts": {"start": "node server.js"}

如果在你的包的根目录有一个server.js文件， 则npm会把node server.js作为默认的开始命令

  • scripts: {"preinstall": "node-waf clean || true; node-waf configure build"}

如果在你的包的根目录有一个wscript文件， 则npm会默认使用preinstall命令去编译使用node-waf

  • scripts: {"preinstall": "node-gyp rebuild"}

如果在你的包的根目录有一个binding.gyp文件,则npm会默认使用preinstall命令去编译使用node-gyp

  • "contributors": [...]

如果在你的包的根目录有一个AUTHORS文件, npm会以Name <email> (url)的格式追踪每一行，email和url是可选的，空行和以#开头的行会被忽略

name

在package.json里最重要的就是name和version字段，它们是必需的，包没有它们是不能安装的，name和version一起形成一个独一无二标示符，包的改变和版本的改变是一起变化的。

这个名称name就是你所需要调用的，一些tips:

  • 不要在名字里放js和name, 它会被假定为js的，因为你正在写的是package.json文件,你能够用engines字段指定引擎（看下面）。
  • 这个名字作为URL结束的一部分，作为命令行的一个参数，作为一个文件夹名称，有non-url-safe字符的任何名字都会别拒绝的，也就是说，它不能以点号或下划线作为开始
  • 这个名字可能作为一个参数给require(),所以它应该是短的，而且也可以被合理描述的。
  • 你可以需要用npm register 看看是否这个name已经被注册了，在你想要提交它之前。

version

在package.json里最重要的就是name和version字段，它们是必需的，包没有它们是不能安装的，name和version一起形成一个独一无二标示符，包的改变和版本的改变是一起变化的。

version必须能够被node-semver—-它是作为npm的一个依赖捆绑的——-解析的（npm install semver to use it yourself）

更多的版本数字和范围参见semver

description

在里面放一个描述，它是字符串，它能够帮助人们发现这个包，在npm search里能被列出来

keywords

在里面放一个关键词，字符串数组，它能够帮助人们发现这个包，在npm search里能被列出来

main

这个mian字段是一个模块ID,这个ID是你程序的最主要入口点，也就是说，如果你的包命名为foo，而且一个用户安装了它，这时候做require(foo)，这是你的mian模块的导出对象会被立即返回。

这个模块ID应该是相对于包根目录的。对于大部分模块来说，设置一个mian字段是很有意义的。

bin

许多包都有一个或更多的想要安装到PATH的可执行文件，npm会使这个变得很重要（事实上，它也是使用这个特性是‘npm’变成可执行的）。

为了使用这个，要提供一个bin字段，这个字段对应的值为一个map, 是命令名到本地文件的一个映射，npm会做一个符号链接添加到profix/bin作为全局安装，或者./node_modules/.bin/作为本地安装。

例如这个：

{”bin“: {"npm": "./cli.js"}}

当你安装npm的时候，会创建一个符号链接，从cli.js脚本到/usr/local/bin/npm

如果你有一个单独的可执行脚本，它的名字要是包的名字，且为字符串：

{ "name": "my-program"

, “version”: “1.2.5”
, “bin”: “./path/to/program” }

和这个是一样的：

{ "name": "my-program"

, “version”: “1.2.5”
, “bin” : { “my-program” : “./path/to/program” } }

repository

指定你的代码存放的地方，这对于想要贡献代码的人是很重要的，如果git仓库放到github上，这时候npm docs命令能够找到它。

就像这样：

"repository" :

{ “type” : “git”
, “url” : “http://github.com/isaacs/npm.git“
}
“repository” :
{ “type” : “svn”
, “url” : “http://v8.googlecode.com/svn/trunk/“
}

这个URL应该是一个公共有效地url,这个能够没有任何修改的直接提交到版本系统里，他不应该是一个能够被浏览器打开的html页面，应该是在桌面使用的url

dependencies

dependencies是以包名为key和版本范围为value的hash对象所指定的。
版本范围是由一个或更多空格分割的描述符所组成的一个字符串。

dependencies也能被一个tarball和git url所指定。

请不要放测试工具和transpilers在你的dependencies hash里，这个请看下面的devDependencise.

可以看semver(7)了解关于版本范围的更多细节。

  • version 必须精确匹配 version
  • >version 必须比 version更大
  • >=version etc
  • <version
  • <=version
  • ~version “大约近似等于这个版本version” 更多了解semver(7)
  • 1.2.x 1.2.0,1.2.1,etc.,但不是1.3.0
  • http://... 看下面的‘URLs as Dependencies’
  • * 匹配所有的版本
  • "“（一个空字符串），等于*
  • version1 - version2 类似于 >=version1 <=version2
  • range1 || range2 两个至少满足一个
  • git... 看下面的‘Git URLs as Dependencies’
  • ‘user/repo’看下面的‘Github urls’

例如：

{ "dependencies" :
    { "foo" : "1.0.0 - 2.9999.9999"
    , "bar" : ">=1.0.2 <2.1.2"
    , "baz" : ">1.0.2 <=2.3.4"
    , "boo" : "2.0.1"
    , "qux" : "<1.0.0 || >=2.3.1 <2.4.5 || >=2.5.2 <3.0.0"
    , "asd" : "http://asdf.com/asdf.tar.gz"
    , "til" : "~1.2"
    , "elf" : "~1.2.3"
    , "two" : "2.x"
    , "thr" : "3.3.x"
    }
}

devDependencies

如果一个人计划在他们的程序里下载和使用你的模块，但他们并想要和需要去下载和build那些额外的测试和文档框架
最好在devDependencies hash里列出这些项。

在包的根目录执行npm link或npm install时这些项能够被安装，也能够被其他任何npm配置参数所管理，更多请参见npm-config(7)

build步骤不是platform-specific,例如编译coffeeScript或其他的语言到javascript，要使用prepublish脚本去做这个。

例如：

json { "name": "ethopia-waza", "description": "a delightfully fruity coffee varietal", "version": "1.2.3", "devDependencies": { "coffee-script": "~1.6.3" }, "scripts": { "prepublish": "coffee -o lib/ -c src/waza.coffee" }, "main": "lib/waza.js" }

这个prepublish脚本会在发布之前运行，以至于用户不用在请求包的时候还要自己编译它们，当然在开发模式（本地运行npm install）, 它还是会很好的运行这个脚本，让你很容易的去测试它
